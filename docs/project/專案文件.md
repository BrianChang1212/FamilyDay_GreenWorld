### 二、核心功能（三項）

**1. 事前報名與統計**（表單，預計 Google 表單或企業維信表單）

- 產出報名清冊（如 Excel）；欄位含工號、姓名、參加人數、**每位參加者**身分證字號與出生年月日（**保險**）。  
- 名額：**1+3** 免費；**第 5 人起**額外收費。  
- 物流：交通（是否搭遊覽車）、午餐桌次等。  
- **待提供**：表單欄位明細、規則說明（人數限制、加價標準、保險必要性等）。

**2. 現場簽到平台**（本專案 Web 核心之一）

- 進場掃**專屬 QR**，輸入工號、姓名並**確認同行人數**。  
- 用途：出席紀錄（**補休假**憑證）、與**餐飲／門票**預約核對；報到禮依**報名人數**發放。

**3. 闖關遊戲**（本專案 Web 核心之二）

- 園內 **6 關**；**實體 QR** 到站（取代 GPS）。  
- 登入綁定工號+姓名；**一工號最多 3 次**／最多 **3 份**闖關紀念品。  
- 題型建議**三選一**或圈選；**答錯可重答至答對**。  
- 完成六關後至**大草皮櫃檯**，工作人員**驗畫面**領獎；**限量、先到先選**。

## 需求與流程

> **最後更新：** 2026-04-10  
> **預估人流：** 約 1,000～1,300 人  
> **系統類型：** Web 應用程式（簽到 + 闖關；事前報名多為表單）

### 一、人數與身分規則

#### 人流規劃
- **預估總人數：** 約 1,000～1,300 人（會議口徑）
- **同時在線：** 規劃需支撐活動日尖峰操作（建議仍依 1,300 級別做壓測假設）

#### 名額、報名與闖關規則

| 項目 | 規則 | 說明 |
|------|------|------|
| **事前報名** | **1+3** 免費；**第 5 人起**加收 | 表單與收費細則由主辦訂定並公告 |
| **簽到禮** | 依**報名人數**領取報到禮 | 與現場簽到紀錄勾稽 |
| **闖關次數／份數** | 同一工號**最多 3 次** | 眷屬分頭或同人重複玩；最多 **3 份**闖關紀念品 |
| **闖關獎項** | **限量**；**先到先選** | 大草皮櫃檯人工作業驗證畫面後領取 |

> **註：** 「3 次／3 份」為會議確認之闖關規則；與**報名加價（第 5 人起）**為不同層次之規則。

### 二、事前報名與現場簽到

#### 2.0 事前報名與統計（表單；多為 Google／企業維信表單）

- **目的：** 活動前統計人數與物流（遊覽車、午餐桌次等）。  
- **產出：** 報名清冊（如 Excel），含同仁填寫之完整資料。  
- **應收集（會議）：** 員工工號、姓名、參加人數、**每一位參加者**之身分證字號與出生年月日（**保險**所需）。  
- **名額：** **1+3** 免費；**第 5 人起**額外收費（標準與說明由主辦於表單載明）。  
- **待主辦提供：** 表單欄位明細、規則說明（人數限制、加價、保險必要性等）。

#### 2.1 現場簽到進場方式

**流程：** 掃描**報到專用 QR code**（或報到連結）→ 應用程式（見 §2.3、§2.4）→ **簽到／報到頁**（原型：`/checkin` **單頁**填姓名／員編／同行人數＋**確認彈窗**；**不**先經闖關用之共用登入頁）→ 報到完成頁 → 領報到禮 → 進入闖關（**另掃**闖關 QR、連結或導覽；見 H4）。  
**報到畫面順序與 Mermaid 流程圖：** [`summary-frontend.md`](../architecture/summary-frontend.md) **§2.2**。

##### 簽到頁欄位（會議）

| 欄位 | 狀態 | 說明 |
|------|:----:|------|
| 工號 | [V] | 必填 |
| 姓名 | [V] | 必填 |
| 同行人數 | [V] | 確認與報名一致，供核對餐飲／門票等 |
| 個人 QR 登記 | [X] | 不使用「每人一組個人 QR」作為登記；進場為**活動／專屬 QR** |

#### 2.2 簽到目的與報到禮

- **出席紀錄：** 作為事後核發**一天補休假**之憑證依據之一（實際以人資規定為準）。  
- **人數核對：** 實際到場與**預約餐飲／門票**等數據勾稽。  
- **報到禮：** 依**報名人數**領取第一份報到禮（與報名清冊勾稽）。

#### 2.3 介面設計（概念）

##### 已確認（方向）

- **實作架構（2026-04-18 補充）：** 簽到與闖關為**同一 Web 應用程式**、**獨立路由**。**原型（`source/`）：** 報到為 **`/checkin` 單頁**（姓名／員編／同行人數＋確認）；闖關身分為 **`/register`**（姓名／員編）。可共用表單樣式元件，但** URL 與欄位**分流（見 [`summary-frontend.md`](../architecture/summary-frontend.md) §2.1）。  
- **進入點（2026-04-18 補充）：** 以**不同 QR code（或不同 URL）** 區分意圖——**掃報到用 QR** → **`/checkin` 報到流程**（或 `/check-in` 寫入意圖後轉址）；**掃闖關入口 QR** → **`/` 歡迎** → 說明 → **`/register`** → 關卡／地圖。若 URL 帶有站點參數（例如某關 JWT／站點代碼），身分就緒後再接續**到站驗證／題目頁**（細節見 [`api-v0.1.md`](../specs/api-v0.1.md) 站點 JWT）。應保留 `entry`、`station` 等 query 或使用 `sessionStorage`，避免流程沖掉導向目標。

```
┌─────────────────────────────────────┐
│   簽到介面（獨立路由／頁面；同一 App）  │
│  • 工號、姓名、確認同行人數           │
│  • 領取報到禮（依報名人數）           │
└─────────────────────────────────────┘
              ⬇️（動線可另掃闖關 QR 或連結）
┌─────────────────────────────────────┐
│   闖關介面（獨立路由／頁面；同一 App）  │
│  • 綁定工號＋姓名；6 關／到站 QR      │
│  • 完成後至大草皮櫃檯驗證領獎         │
└─────────────────────────────────────┘
```

#### 2.4 QR 進入點與意圖分流（2026-04-18 補充）

> **註：** 舊稿「共用登入」多指闖關 **`/register`**；**報到**原型為 **`/checkin` 單頁**（§2.1、[`summary-frontend.md`](../architecture/summary-frontend.md) §2.1）。

| 項目 | 說明 |
|------|------|
| **目標** | 現場可印製**至少兩類**入口 QR：**報到**、**闖關**（闖關亦可再細分「總入口」與「各關到站 QR」，後者見 §3.1）。使用者掃碼後進入**同一網域／同一前端專案**，依 URL 參數或路徑決定登入後預設畫面。 |
| **報到 QR** | 連結指向例如 `/check-in?...` 或 `/?entry=checkin`（實際路徑以實作為準）。**原型（`source/`）：** 寫入意圖後 **`/checkin` 單頁**（含確認彈窗），**不**先走與闖關共用之 `/register` 登入頁。 |
| **闖關 QR** | 連結指向例如 `/game?...` 或 `/?entry=game`；**原型（`source/`）：** `/game` 寫入意圖後導向 **`/` 歡迎**，再 **遊戲說明 → `/register` → 地圖／關卡**。若為某關到站，可附加站點 token／參數供後端驗證，身分就緒後接題目流程。 |
| **與 API 草案** | 簽到與闖關之 **REST 端點分開**（見 [`api-v0.1.md`](../specs/api-v0.1.md)）；前端可共用**表單樣式元件**，並實作 **QR／意圖分流與路由**（報到 **`/checkin`**、闖關 **`/register`**，見 §2.1）。 |
| **待產品定案** | **未報到是否允許進闖關**（建議：不允許時由路由守衛或後端拒絕；允許時需主辦明確同意）。 |

##### 待確認

### 三、闖關與獎品機制

#### 3.1 關卡設計（0410 會議）

| 項目 | 規格 |
|------|------|
| **總關卡數** | 6 關（六大主題區；**5 月下旬場勘**後與園方確認布點） |
| **到站機制** | 各關**實體 QR code**（取代 GPS，減少誤差） |
| **題型** | 建議**三選一**或圈選題；簡單有趣、鼓勵參與 |
| **作答規則** | **答錯可立即重答，直到答對為止**（會議口徑） |
| **每關題數** | 舊稿曾列每關 3 題；**可與題庫設計併案**，以不超過現場體驗負擔為原則 |

##### 闖關流程（概念）

```
第1關 → … → 第6關 → 完成（後端紀錄可供櫃檯驗證）
各關：掃描該關實體 QR → 答題（錯則同關重試至通過）→ 進下一關
```

#### 3.1.1 闖關畫面順序（掃描**闖關入口** QR／連結之後 · 線框對齊）

> **範圍：** 以下為**闖關動線**專用；**不含**現場報到（報到另掃報到 QR，見 §2）。資料來源：闖關遊戲流程規劃線框（2026）。  
> **與報到之差異：** 闖關線在遊戲說明後進入 **闖關登入頁**，僅填 **姓名／員工編號**；**同行人數**僅於報到流程收集。

**（一）進入遊戲後 → 開始闖關前**

| 順序 | 畫面 | 重點 |
|:----:|------|------|
| 1 | **歡迎頁** | 活動主視覺與標題（如「歡迎來到 2026 瑞軒家庭日」）；主按鈕「開始探索」進入說明。 |
| 2 | **遊戲說明** | 標題「遊戲說明」；**闖關說明**（完成 6 大關卡可領闖關禮）；**關卡地點順序**（線框示例：天鵝湖 → 大探奇區 → 水生植物公園 → 鳥園 → 蝴蝶園 → 生物多樣性探索區；實際以場勘為準）；**注意事項**（例：1 員編最多登入 3 台裝置、最多領 3 份獎品）；主按鈕「開始探索」→ **下一屏即闖關登入頁**（不先進地圖或題目）。 |
| 3 | **闖關登入頁** | **呈現方式：** **全屏表單頁**（線框與流程圖上之「**登入**」步），**非**彈窗（Modal）。**文案：** 標題常為「請輸入您的基本資料」。**欄位：** **姓名**、**員工編號**；**無**同行人數。**動作：** **「確定」** 後送後端／綁定身分，再進入關卡流程（**開始闖關**）。頁內或旁註提醒：資料與兌換闖關禮、裝置／獎品上限有關。 |

**（二）每一關（可循環至第 6 關）**

| 順序 | 畫面 | 重點 |
|:----:|------|------|
| 1 | **到站／關卡頁** | 顯示本關名稱；引導抵達後掃描站點 QR；**相機**開啟掃描介面。 |
| 2 | **掃描 QR** | 取景框與提示；**掃描成功**後**直接進入題目頁**（不另繞路）。 |
| 3 | **題目頁** | 闖關進度（如 01/06）；題幹與選項；**未選擇答案前「確定」維持不可按**；選取後可送出。 |
| 4a | **答錯** | 錯誤提示與鼓勵文案；**重新回答**回到**同一題**（或同一關題目狀態）。 |
| 4b | **答對** | 成功提示；**前往下一關**回到類似「到站頁」結構，進度遞增。 |

**（三）六關完成後 → 領闖關禮**

| 順序 | 畫面 | 重點 |
|:----:|------|------|
| 1 | **完成闖關** | 如「恭喜完成闖關」、顯示姓名／員編；引導至**統籌站／指定處**領取闖關禮；可含獎項／點數圖示（未領前為未啟用狀態）；主按鈕「領取闖關禮」等；附註請**工作人員**協助核銷。 |
| 2 | **確認領取（彈窗）** | 如「確認領取？」、第幾次領取、確認後無法取消；**確認領取**／**取消**。 |
| 3 | **領取成功** | 成功文案與感謝；獎項圖示狀態更新（已領項目視覺區隔）；與會議「**3 次／3 份**」及櫃檯驗證並列考量（詳 [`api-v0.1.md`](../specs/api-v0.1.md)；前端 **`/finish`** → **`/finish/claimed`**，`ClaimSuccessView` 以 **`GET /api/v1/me/dashboard`** 之 `progress` 呈現次數，無 API 時 **`local-fallback`**，見 [`summary-frontend.md`](../architecture/summary-frontend.md) §2.1、§2 目錄 **v1.26**）。 |

### 四、技術規格

#### 4.1 登入與驗證

- **簽到進場：** 掃描**報到專用 QR code**（或活動方提供之報到連結）進入應用程式；**原型**為 **`/checkin` 單頁**（姓名／員編／同行人數＋確認）→ 報到完成頁（非每人一組個人 QR 登記）。若未來與後端「先 auth 再簽到」合併，仍以 §2.2 線框欄位為準。  
- **闖關：** 掃描**闖關入口 QR** 或一般連結進入後，畫面順序以 **§3.1.1** 為準：**歡迎 → 遊戲說明 → 闖關登入頁（全屏表單：姓名／員編，非彈窗）→ 各關到站與掃碼答題 → 完成與領獎**。各關以**實體站點 QR** 驗證到站後進入題目（URL 需保留站點參數／JWT，見 §2.4、[`api-v0.1.md`](../specs/api-v0.1.md)）。  
- **身分欄位（闖關線）：** 於**闖關登入頁**填寫 **工號＋姓名**（與會議一致）；**不含**報到專用之同行人數（見 §2）。  
- **報到線**與**闖關線**入口 QR **分開**；報到完成**不**自動進入闖關（見 §2.3、§2.4）。

#### 4.2 效能需求

| 指標 | 規格 |
|------|------|
| **預估同時在線人數** | 約 1,000～1,300 人（會議口徑） |
| **系統支援** | 規劃上仍以 **1,300 人級**尖峰操作為壓測參考（簽到、闖關、核銷） |
| **待確認** | 尖峰流量時段規劃與負載測試需求 |

##### 技術細節（狀態：草案已寫入補充文件，待簽核）

**建議草案（2026-04-10 架構討論，詳見 [`summary-*.md`](../architecture/summary-frontend.md)）：**

- **前端：** Vue 3 + Vite + TypeScript + Tailwind CSS + **Vue Router**（**實作**）；**Naive UI** 仍為草案選項、尚未安裝；表單與元件以 Tailwind 自製。簽到／闖關為**不同路由／模組**：**報到**為 **`/checkin`**，**闖關身分**為 **`/register`**（見 [`summary-frontend.md`](../architecture/summary-frontend.md) §2.1）；**QR／URL 進入點**寫入意圖並分流（§2.4）。關卡瀏覽仍以草案 **`GET /api/v1/me/dashboard`** 單次載入為目標；無操作時不發背景請求。  
- **後端：** 採 **Firebase**（Cloud Firestore 為主，Realtime Database 視即時連線需求啟用）；站點 QR 內容仍採 **signed JWT** 驗證；簽到與闖關登入流程分開；進度模型維持 **作法 A**（無獨立 `runId`）。  
- **Sheet：** 以報名清冊匯入／報表匯出為主；**即時闖關與簽到寫入**以 DB 為準（避免 Sheets API 配額瓶頸）。  
- **RWD：** 活動現場以手機為主，**採用響應式**（草案預設需要）。

### 關鍵決策記錄

| 日期 | 決策 | 理由 | 影響 |
|------|------|------|------|
| 2026-04-10 | 採用 Web UI 而非 App | 跨平台、無需安裝、開發成本低 | 降低使用門檻，加快開發速度 |
| 2026-04-10 | 後端採用 Google Sheet + Database 混合架構 | Sheet 方便人工管理，DB 處理高併發 | 兼顧管理便利性與系統效能 |
| 2026-04-10 | 報到與闖關分為兩個獨立介面 | 流程清晰，功能分離 | 降低單頁複雜度，提升使用體驗 |
| 2026-04-18 | 同一 Web App 內**共用登入**；**報到 QR** 與**闖關 QR** 分流入，登入後導向對應模組 | 減少重複填寫；現場可區分進場意圖；與 wireframe「點連結或掃 QR」一致 | 前端需實作進入點保留與登入後 redirect；**未報到能否進闖關**仍待產品定案（§2.4） |
| 2026-04-10 | 事前報名與現場系統分離 | 表單（Google／維信）產清冊；Web 負責簽到與闖關 | 開發範圍與資料邊界清楚 |
| 2026-04-10 | 闖關到站以實體 QR 取代 GPS | 減少定位誤差 | 須場勘確認布點與防偽 |
| 2026-04-27 | **後端架構定案**（Vue+Vite+Firebase、API v0.1、流量與部署摘要） | 與會議成本估算及維運策略對齊，降低後端維運負擔 | 見本檔開頭補充文件表；後端以 Firebase 為準 |
| 2026-04-15 | **`source/` 前端可建置原型**（Vue Router、Tailwind、示範流程） | 驗證技術鏈與 UX 假設；與後端解耦 | Naive UI 未裝；見 `summary-frontend.md` v1.1 |
| 2026-04-16 | **完成頁 `/finish` 領獎流程（原型）** | 對齊「3 次／3 份」與櫃台驗證情境；先以瀏覽器暫存類比領取次數 | 見 `summary-frontend.md` v1.2（**現況**見 **v1.26**、`api-v0.1` **v0.1.8**）；正式上線需接 `api-v0.1.md` 櫃台／核銷端點或等價流程 |

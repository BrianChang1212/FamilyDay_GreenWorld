# 流量與容量 — 討論總結

> 本文件彙整**在線人數、請求行為、尖峰估算、限流與容量規劃**之討論結論。  
> 與前端行為呼應見 [`summary-frontend.md`](./summary-frontend.md)；後端實作見 [`summary-backend.md`](./summary-backend.md)。

---

## 1. 核心結論（必讀）

| 觀念 | 結論 |
|------|------|
| 在線人數 vs 伺服器負載 | **1,500～2,000 人同時在線**不代表固定高 RPS；若**不刷新、無輪詢**，**無操作即幾乎無請求**。 |
| 容量設計主軸 | 應以 **「尖峰時每秒實際 API 請求數」** 與 **資料庫寫入** 為主，而非僅看在線人數。 |
| 極端假設的澄清 | 若誤用「每人每分鐘讀寫各 60 次且全員打滿」估算，會得出 **每秒數千次** 等級，**與本產品互動模式不符**；已改以實際流程重估。 |

---

## 2. 產品假設下的請求行為

依討論中之使用者流程（摘要）：

1. 掃 QR 進登入頁：**約 1 次**（或含進場驗證則略增）。  
2. 輸入工號／姓名：**每次送出 1 次**；失敗重試受 **每分鐘 30 次／人** 限制。  
3. 登入成功進關卡瀏覽：**`GET /api/v1/me/dashboard` 1 次**；手動刷新則再打，同受 **30 次／分鐘／人**。  
4. 點關卡 → 掃站點 QR → 進關卡頁 → 作答：以 **verify + 載題 + 送答案** 為主，答錯重送；整段受 **30 次／分鐘／人**。  
5. 共 **6 關**；可回關卡瀏覽；全數完成後進成功頁。  
6. **無自動刷新**：閒置不產生背景流量。

---

## 3. 快樂路徑請求量（每人、一輪）

粗估（實際依 API 切分正負幾支）：

- 登入相關：**約 1～2 支**  
- 關卡瀏覽：**1 支**（`GET /api/v1/me/dashboard`）  
- **每關**：站點驗證 + 載題 + 答對（約 **2～3 支／關**）  
- **6 關**：約 **12～18 支**  
- 回瀏覽、成功頁：**約 1～2 支**  

**合計一輪完整遊玩：約 15～25 支 API／人**，且分散在**數十分鐘以上**，**平均 RPS 很低**。

---

## 4. 尖峰情境（數量級）

| 情境 | 說明 |
|------|------|
| 整點／進場集中 | 短時間內大量 **登入** 或 **dashboard**；例如若 1 秒內 500 人各 1 請求 ≈ **500 RPS 瞬間**（僅為舉例，實際會隨時間攤平）。 |
| 分散闖關 | 多人分散在不同關，請求較平緩。 |
| 惡意或連續錯誤 | **30 次／分鐘／人** 上限使單人最大約 **0.5 req/s**；全員同時打滿屬極端假設，實務上罕見。 |

整體在**無輪詢**前提下，以 **Firebase（Firestore 為主）** 的讀寫模型多可負荷；仍須 **壓測驗證**。

---

## 5. 限流策略（與後端一致）

- **每使用者每分鐘 30 次**（可區分：登入桶、一般桶，避免刷新用光額度導致無法答題）。  
- **伺服器端**強制執行；前端配合 **429** 與提示。  
- 登入／簽到可**獨立更嚴格**限流。

---

## 6. 容量與平台建議（討論摘要）

- **依目前定案**：後端以 Firebase 為主，重點轉為 **讀寫熱點、查詢路徑、下載流量** 管控。  
- **即時場景**若啟用 Realtime Database，需額外監控同時連線與傳輸量。  
- **必做**：以 **k6**（或同等工具）模擬 **(A) 登入／dashboard 尖峰**、**(B) 多關作答混合**，觀察 **p95 延遲、錯誤率、讀寫與流量指標**。

---

## 7. 與 README 預估人數的關係

- README 載活動預估約 **1,000～1,300** 人；討論中曾以 **1,500～2,000 在線** 作保守在線上限，**不改變「操作驅動、低背景流量」結論**。  
- 若實際到場更高，仍以 **壓測 + 當日監控** 為準。
